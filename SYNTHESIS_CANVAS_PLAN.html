Synthesis Canvas
selection-scoped generation for During Workshop

Your vision: Requirements, Capability Map, Process Flow and the BRD should be created from documents the facilitator deliberately drags onto a small canvas — not from "everything in the workshop" — with an optional prompt steering each run.

01 · The concept

A workbench, not another whiteboard

The "small canvas" is a staging area: the facilitator composes a working set of inputs, optionally types an instruction, and fires a generator. Everything downstream is scoped to exactly that set.

Generate: ≡ Requirements ◎ Capability map ⛓ Process flow ▤ BRD ✦ Auto — let the router pick from the prompt

The flow, end to end

  1. Compose — drag rows from the Artifact Explorer (sources and generated docs both allowed; generated inputs stay provenance-marked). Click-to-add works too (a small “+ to canvas” on each explorer row) so it's not drag-only.
  2. Steer — optional prompt, passed as the agent's EXTRA INPUT (the plumbing every agent already has).
  3. Generate — pick a generator, or hit Auto and a cheap router call (same pattern as Copilot dispatch) classifies the prompt to the right one.
  4. Trace — the run is scoped server-side to the selected doc ids; the output artifact stores inputs_json (which documents built it), shown on its card as “built from 3 inputs”.
  5. Iterate — the working set persists (per workshop, sessionStorage), so the natural sequence “same set → requirements → capability map → BRD” is three clicks.
02 · One agent or many?

Keep the four specialists, add one shared mechanism

Your question answered directly:

  • Keep the four existing agents (extract_reqs, capmap, workflow, brd). Each has a strict, different output contract (table rows vs. heat-map JSON vs. diagram JSON vs. long document) with its own validation and persistence — merging them into one agent means one giant prompt doing four jobs worse, and one parse failure killing all four outputs.
  • What's genuinely shared is the scoping, so build it once: a doc_ids option on run_agent that every context builder respects. The canvas doesn't need new agents — it needs the existing agents to accept “use only these inputs”.
  • The “single agent” feel comes from the Auto button: one router call (the cheap BEDROCK_ROUTER_MODEL_ID, same as Copilot's dispatcher) reads the prompt and picks the generator. User experiences one smart agent; quality stays specialist.
Design principle Selection is a context rule, not an agent. Agents stay experts in their output; the canvas decides what they're allowed to read.

06 · Decisions & edge cases

To confirm before building

  • Strict gating — plan follows your rule: empty canvas = generators disabled, with an explicit “add all” chip as the whole-corpus path. Alternative: empty canvas silently means “everything” (today's behaviour). Which do you want?
  • Generated docs as inputs — plan allows it (research briefs are legitimate BRD inputs), always provenance-prefixed. Veto if you want sources-only.
  • Auto button in v1 — include, or ship the four explicit buttons first and add Auto in M3? (Plan: M3.)
  • Requirements dedup stays workshop-global — extracting from a selection still won't duplicate an existing REQ row captured earlier from another document. Assumed yes.
  • Pre-Workshop parity later — the same canvas could scope deepresearch/analyze in Pre-Workshop. Out of scope here; the mechanism will already support it.
Honest boundary The canvas is a staging strip, not a spatial whiteboard — chips don't have meaningful x/y positions. If you later want arrangement to mean something (e.g. ordering inputs by priority), that's a separate design.
