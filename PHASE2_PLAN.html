03 · The centerpiece

Why native SVG, not an embedded iframe

  • We already own the data. Every workflow diagram persists as structured JSON (diagram_json: lanes, typed nodes, labeled edges) — rendering it ourselves is a pure-frontend exercise with zero new backend work.
  • Full visual control. Lane tints, node shapes, hover states, highlight-on-select, entry animations — none of which an embed.diagrams.net iframe allows. The iframe is also heavy (~3–5s load) and needs internet.
  • The editor stays. “Edit in draw.io” opens our existing editor modal with the same XML; Export downloads the same file. View natively, edit externally — each tool where it's strongest.

05 · New backend

Three agents and one table

Requirements engine

  • Table requirements: req_id (REQ-01…), workshop_id, text, category, moscow (must/should/could/wont), source_label, source_doc_id, status, sort order — plus repo + CRUD routes (list/create/update/delete). Editable and manually extendable (“+ Add requirement”), not just AI output.
  • Agent extract_reqs: reads imported transcripts + pre-workshop context (the cached corpus — the reference explicitly “fuses pre-workshop context with capture”); returns a strict-JSON array of requirements with source attribution per item; upserts with stable IDs so re-running after a new import adds instead of duplicating.

Capability map

  • Agent capmap: structured contract {domains:[{name, capabilities:[{name, maturity 1–5, opportunity high/med/low, note}]}]} — same coerce/clamp discipline as insights/gaps; persisted on the generated doc (a capmap_json column, mirroring analysis_json) so the panel survives reload; versioned by re-run (v1.0 → v1.1).

BRD assembler

  • Agent brd: composes the approved requirements table + capability map + engagement context into the formal BRD document — persisted, Word-exportable, RAG-indexed like every generated doc. Card shows “N reqs · M capability areas”.
Stats endpoint One small GET /api/agents/workshop-stats aggregates the hero chips (transcripts imported, requirement count, capability count, live artifacts) instead of four client-side fetches.

08 · Boundaries & open questions

Deliberately out of scope

  • Live transcription / live Teams capture — your explicit call; requires an Azure bot-registration project (assessed previously as multi-week).
  • In-canvas diagram editing — the viewer is read-only; editing delegates to the proven draw.io modal (see §03 “Honest scope”).
  • Granola / GitHub / whiteboard-photo ingestion — the reference shows these source types; our sidebar will show correct type icons for anything uploaded (images, links as text), but real Granola API and repo-scanning integrations are separate efforts. Whiteboard photo → text needs an OCR/vision step we don't have yet.
  • Real-time multi-user sync — panels refresh by polling (the pattern already used for research/analysis progress), not websockets.

To confirm before/while building

  • Retire the During Workshop canvas entirely, or keep it reachable (e.g. a “Board view” toggle) during transition? Plan assumes full replacement, matching what we did for Pre-Workshop.
  • MoSCoW edits and requirement approvals — single-editor assumption OK for now (no locking)?
  • Should extract_reqs run automatically on every transcript import, or stay behind an explicit “Extract requirements” action? Plan assumes explicit first, auto later.
