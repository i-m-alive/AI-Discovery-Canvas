5Generation pipeline

Extends the existing agent_catalog.py::run_agent pattern — no new LLM plumbing.

New agent: backlog

  • Reads: approved requirements rows + capability map (capmap_json) if present.
  • One structured LLM call, JSON schema: { epics: [{ title, features: [{ title, stories: [{ text, acceptance_criteria }] }] }] } — same shape/coercion approach as capmap's domains→capabilities nesting.
  • Sanitize → replace-on-regenerate into backlog_epics/backlog_features/backlog_stories (delete-and-reinsert per workshop, matching how other regenerable docs behave), assigning stable EPIC-*/FEAT-*/US-* ids.
  • Retires the old flat stories and bdd agents' role in Post-Workshop — their prompts get merged into this one generator rather than running as three disconnected drafts.

Reshaped agent: opportunities

  • Reads: requirements + any explicit "pain point"/"parking lot" tagged content from the During-Workshop capture.
  • Structured output: { opportunities: [{ title, description, horizon, size, priority }] }.
  • Flip doc: False → True-equivalent by persisting rows into the new opportunities table.

Reshaped agent: mom

  • Context builder extended to include imported Teams transcripts (from During-Workshop's transcript import, already built) + Granola notes + manually-added BA notes.
  • Output reshaped to { decisions: [...], actions: [{owner, text, due}], open_questions: [...], confidence_pct }, persisted to the new mom_json column.
  • Confidence score: derive from source coverage (e.g. % of discussion segments with a matched transcript timestamp) rather than asking the model to self-rate — self-rated confidence is unreliable.

Frontend generation trigger

  • Refactor SynthesisCanvas.jsx to accept a generators prop instead of importing the module-level GENERATORS constant — small, low-risk change, and During-Workshop keeps working unchanged by passing its existing array explicitly.
  • Post-Workshop passes [{id:'backlog',...},{id:'opportunities',...},{id:'mom',...}] and reuses the same dropzone → chips → run → pipeline-status-pills → result-banner interaction.

7Azure DevOps / Jira sync

Scope call needed Your reference design's banner says "Two-way sync keeps this backlog aligned with your board" — true two-way sync (pulling back edits made on the client's actual ADO/Jira board, resolving conflicts) is a significantly bigger lift than one-way push (create/update work items from this app). Recommend building one-way push first and treating true bidirectional sync as a later phase — see Decisions Needed.

Azure DevOps

  • Auth: the existing source_oauth.py ADO OAuth entry already requests the correct resource scope (499b84ac-1321-427f-aa17-267ca6975798/.default, the Azure DevOps resource ID) — the token-exchange plumbing is reusable even though it was built for git cloning, not Work Items.
  • Push via PATCH https://dev.azure.com/{org}/{project}/_apis/wit/workitems/$<type>?api-version=7.1 (JSON Patch body) — Epic → work item type Epic, Feature → Feature, Story → User Story (or Product Backlog Item, depends on the client's process template — needs to be configurable per workshop/project).
  • Parent-child links via the System.LinkTypes.Hierarchy-Reverse relation field.
  • Acceptance criteria go into the story's Microsoft.VSTS.Common.AcceptanceCriteria field.
  • New service: backend/app/services/ado_workitems.py.

Jira

  • Auth: simplest viable v1 is Jira Cloud API token + email (HTTP Basic) rather than 3-legged OAuth — faster to ship, fine for a single connected account per engagement.
  • Push via POST /rest/api/3/issue — Epic → issue type Epic, Feature → mapped to a custom issue type or a labeled Story (Jira's default scheme has no native "Feature" level — needs a per-client mapping decision), Story → Story, linked to its epic via fields.parent.
  • Acceptance criteria go into the issue description or a custom field, depending on the client's Jira configuration.
  • New service: backend/app/services/jira_workitems.py.

Idempotent re-push

Every pushed item gets a row in backlog_sync_links with a content_hash of the fields last sent. "Push N items" only counts items that are new or whose hash changed since last sync — this is what makes repeated pushes safe rather than creating duplicate work items every time.

9Decisions needed from you

  • Sync direction: ship one-way push first (recommended) and label the UI accordingly, or commit to true two-way sync from the start (bigger scope: webhooks, conflict resolution)?
  • ADO auth: reuse OAuth (better UX, more setup) or a simpler Personal Access Token field per project for v1?
  • Jira auth: API token + email (fast to ship) or full OAuth 2.0 3-legged flow?
  • Jira Feature mapping: Jira has no native "Feature" level between Epic and Story by default — map Features to a custom issue type, or flatten Epic → Story and drop the Feature tier for Jira pushes specifically?
  • Multi-day workshops: your reference shows "Minutes of Meeting — Day 1" — do engagements commonly span multiple workshop days needing separate MoM cards per day, or is one consolidated MoM per workshop enough for now?
  • Facilitator editing: should generated backlog items be editable in place before sync (recommended — LLM output should get a human pass before hitting a real board), or generate-and-push directly?
  • "X/Y ingested" counter copy: keep ArtifactExplorer's current "X source(s) · Y generated" phrasing, or restore the "ingested" ratio wording from the dead SourceArtifactsPanel component to match your reference screenshot exactly?
