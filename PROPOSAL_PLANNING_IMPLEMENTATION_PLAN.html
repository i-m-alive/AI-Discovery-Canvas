4Data model — one additive column, no new tables

Unlike the backlog (per-item edits + per-item sync state → normalized tables), all four proposal artifacts are whole-document payloads regenerated as a unit and lightly edited in place — exactly the capmap_json/mom_json shape. One generic column keeps it to a single migration line; each generated_docs row is produced by exactly one agent, so the payload shape is unambiguous per row.

generated_docs.proposal_json  JSONB   — shape keyed by the producing agent:

'sow':  { engagement_weeks: 20,
          milestones: [{week: 2, title: "Mobilization & discovery sign-off"}, …] }

'roi':  { currency: "£", horizon_years: 3,
          net_value: 4.6, net_value_label: "£4.6M", payback_months: 14,
          series: [{period: "Year 1", cumulative_value: 0.9, cumulative_cost: 1.1}, …],
          drivers: [{name: "Overtime reduction", pct: 42}, …],
          basis: "one line on what the estimate is grounded in" }

'risk': { items: [{label: "Legacy scheduling data quality",
                   benefit: 62, risk: 78, note: "…"}, …] }

'team': { roles: [{role: "Engagement Lead / BA", count: 1,
                   allocation_pct: 50, why: "…"}, …] }
Migration One line added to bootstrap_postgres's _ADDITIVE_COLUMNS: ('generated_docs', 'proposal_json', 'JSONB') — idempotent, runs on backend restart. Plus a proposal_json kwarg threaded through generated_docs.register() → repo.create() and a latest_proposal(workshop_id, agent_id) reader (same shape as latest_mom), and a has_proposal flag in list_docs.

5Generation pipeline

Four reshapes of the proven pattern — structured contract → coercer → persistence — plus one new shared context builder. No new LLM plumbing.

New shared context: _proposal_context

  • Captured requirements table (as text, like _engagement_context).
  • Latest capability map (maturity/opportunity per capability — feeds ROI drivers and risk items).
  • The backlog tree — epics/features/story counts formatted as text; milestones align to epics, team sizing reads scope breadth.
  • Accepted opportunities from the register (phase-1 items may enter scope).
  • Corpus distillations + canvas selection override, same as every other phase.

sow — reshape

  • _SOW_FIELD: milestones array (3–8 items, week numbers fitted to the engagement length from EXTRA INPUT, defaulting to ~20 weeks).
  • Coercer clamps weeks to 1–104, titles to 120 chars, sorts by week.
  • Persist to proposal_json; keep prose body_html (the full SOW document) unchanged.

roi — reshape (biggest one)

  • _ROI_FIELD: net value + label, payback months, per-year cumulative series, 3–6 drivers summing ≈100%, and a mandatory basis line naming the client facts the estimate stands on.
  • Honesty rule carried over from the existing task: sparse context → wide-range or null numbers with the basis saying what data would firm them up. The UI renders "—" tiles rather than fabricated certainty.
  • doc: False → True; category "ROI".

risk + team — reshape

  • _RISK_FIELD: 3–8 scored items (benefit/risk 0–100, evidence note each). Scores must trace to material facts, not generic risk-register filler.
  • _TEAM_FIELD: 4–8 roles with count, allocation %, one-line why tied to THIS scope.
  • Both doc: False → True so they persist, appear on the grid, and export.
  • Both join run_agent's context branch: elif agent_id in ('sow','roi','risk','team'): context = _proposal_context(…); sow/roi join the 8000-token output tier.

6Charts — inline SVG, zero dependencies

The repo has no charting library and shouldn't gain one for three small charts. Each is a self-contained React component rendering plain SVG, themed via the existing CSS variables so dark mode works for free:

  • RoiChart.jsx — two polylines + one gradient-filled area (cumulative value vs delivery cost), x = periods, y auto-scaled, dotted crossover gridlines, legend dots. ~80 lines.
  • RiskScatter.jsx — 0–100 × 0–100 plot, quadrant gridlines at 50, labeled circles, "Top-left = high benefit, low risk (prioritize)" footnote. ~70 lines.
  • MilestoneTimeline.jsx — evenly-spaced flag nodes on a connector line, week label + title under each; horizontal scroll inside the panel when narrow. ~50 lines.
  • Value-driver bars need no SVG — a div-based percent bar per driver.

Precedent: DiagramCanvas.jsx already renders the During-Workshop process flows natively — hand-rolled visuals are this codebase's established approach.
