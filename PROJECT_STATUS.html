1. Where we actually are, honestly

The product loop works end to end: a Business Analyst can open the canvas, move between Prepare/Run/Synthesize/Project, run any of 18 real AI agents against real board content, transcript, and uploaded documents, approve/edit/reject the draft, watch it file into the artifact library with provenance, and export a real DOCX or Markdown handoff package. Deep research, draw.io workflow diagrams, live microphone transcription, and a Teams transcript pull (on NaviCore's real Entra app registration) are all wired and passed their tests. That is genuinely most of the product surface from the Master Documentation.

But "the code path exists and passes mocked tests" is a different claim from "this is production-ready," and I want to be precise about which one is true where. Nothing has been tested against a live Bedrock model yet (no credentials are configured) — every agent, the copilot's intent router, and deep research all currently return the same one error path (BEDROCK_MODEL_ID is not set), which is correct behaviour but means the actual quality of model output is completely unverified. There is also no automated test suite living in the repository — every check I've described across four phases was a script I ran once in this session, not a pytest file you or a CI system can re-run. And several structural gaps (single shared board, no real-time collaboration, guardrails copied but never invoked) are the kind of thing that's fine for a demo and a real problem for a second user or a regulated client. Section 4 is the honest list.

4. What's missing that I haven't flagged clearly before

Some of these were mentioned in passing during earlier phases; collected here because they compound into a real "not production ready" picture that's easy to miss when each phase report says "smoke tests passed."

Testing & safety net

  • Zero committed automated tests. Every "8 tests passed" report in this conversation was a script typed into a Python heredoc and thrown away. Nothing stops a future prompt/route change from silently breaking auth, the sanitizer, or the export pipeline.
  • No CI. No GitHub Actions/equivalent — nothing runs even the manual tests automatically on push.
  • The sanitizer's threat model is model output only. It correctly strips script/attributes from what Bedrock returns, but there's no adversarial testing against a genuinely hostile uploaded document trying to jailbreak the "treat as data not instructions" framing.

Data & multi-tenancy

  • One board for the whole app (Section 3, FR-12) — no concept of "engagement" or "client" yet, despite the NFR requiring per-engagement data scoping.
  • Neo4j is wired but unused. The auth/RAG/graph client code all still points at Neo4j, but it isn't even running right now (verified: port 7688 refuses connections) and the canvas board lives in a flat file instead. Fine for one demo board; not a real data model.
  • Guardrails package (PII/secret masking) is copied but never called. Every uploaded document goes straight into prompts and the RAG index unmasked. This matters a lot the moment a regulated client's real document touches this system.

Security & ops hardening

  • AUTH_JWT_SECRET is empty → a random secret is generated per process, so every backend restart invalidates every session. Fine for dev, breaks any real deployment.
  • No per-user rate limiting or cost controls on Bedrock calls — the existing limiter is a process-wide RPM/TPM safety valve, not a per-user quota or spend cap. Nothing stops one user (or a bug in a retry loop) from running up a real AWS bill.
  • No Dockerfile for this backend, no chosen hosting target, no deployment story at all. Everything runs on your laptop.
  • CORS_ORIGINS in .env still says :3000 but the frontend has been running on :3002 this whole time (port 3000 is taken by your other project). Harmless today because the Next.js rewrite keeps everything same-origin from the browser's perspective, but it's config drift that will bite if anything ever calls Flask directly.

Product surface

  • PDF export is a UI placeholder ("coming later") — only DOCX and Markdown actually download.
  • Granola is intentionally a stub (your call) — flow exists, no live account.
  • No mobile/responsive layout — desktop-only, same as the prototype, but worth naming since nobody has asked "should a BA be able to open this on a tablet in the room."
  • No accessibility pass — no ARIA labelling, no full keyboard navigation beyond Delete/Cmd+D, no screen-reader consideration.
  • No React error boundary — an unexpected JS exception inside the ported canvas engine would currently blank the page rather than show a recoverable error.

6. Recommended next steps, in order

Right now — unblock real verification

~30 min of your time

  • Do one real Teams test: connect, paste a join URL from a meeting that had transcription on, confirm lines import.
  • Re-click Search and Presentation Mode once, just to close Section 5's open item.

Near-term — make the safety net real

3-5 days
  • Turn this session's ad-hoc test scripts into a committed backend/tests/ pytest suite (they already exist in spirit — 20+ assertions across four phases) plus a minimal GitHub Actions workflow.
  • Wire the guardrails package into the upload path (mask PII/secrets before anything reaches a prompt or the RAG index) — this is a small integration, not new capability, since the package is already copied and working.
  • Fix the CORS_ORIGINS/port drift; set a real AUTH_JWT_SECRET.

Medium-term — the two real architectural gaps

2-3 weeks
  • Multi-engagement data model: move off the single 'default' board to per-engagement boards (this is exactly what the already-wired-but-unused Neo4j layer is for — get it running and give each board a real identity).
  • Real-time multi-user (FR-12): today it's decorative avatars; making it real needs a WebSocket/SSE layer and a conflict-resolution strategy (even a simple "last-write-wins per node" would beat today's "last-write-wins per whole board").
  • Pick and stand up a real deployment target (Dockerfile + hosting decision) once the above two are underway, so you're not hardening a design that's about to change.

Whenever it's warranted

on demand
  • PDF export, once DOCX/Markdown have been used for real and you know if PDF is actually needed.
  • Granola, only once the subscription decision is made (per your instruction).

7. Ideas worth considering (not requested, not started)

None of these are asked-for — flagging them because they came up naturally while building the parts that were requested. Worth a quick "yes/no/later" rather than being silently assumed.

  • A visible Bedrock cost/usage counter in the assistant panel — every agent call has a real per-token AWS cost, and right now the facilitator has no visibility into it at all.
  • Undo/version history for the canvas — right now Reject discards a draft, but there's no way to undo an accidental Approve or a deleted node short of not having saved yet.
  • A lightweight admin/engagement picker — becomes necessary the moment multi-board support exists (Section 6); worth designing at the same time rather than bolting on after.
  • Template pre-seeds per industry (pharmacovigilance, banking, retail...) for the Prepare zone — this was already a "Phase 3 candidate" in the original Master Documentation roadmap and fits naturally once a few real engagements have run through the system.
  • ADO-importable JSON export for stories/acceptance criteria — named explicitly in the original spec as a Phase 2 integration item; not started, would slot in next to the DOCX/Markdown export.
