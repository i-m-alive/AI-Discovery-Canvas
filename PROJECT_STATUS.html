3b. Phase 2 (During Workshop) — feature plan, before building

Requested before implementation starts: what the reference shows for this phase, what we can already reuse from the Pre-Workshop build, and what's genuinely new.

Live Teams transcription — verdict: not realistically achievable right now. Microsoft Graph has no subscribable live-caption feed for a normal web app. The only real path is registering this app as an actual Azure Bot Framework calling bot that joins the meeting as a participant and receives raw media — requiring the gated Application permission Calls.AccessMedia.All (far harder to get consented than the scopes already stuck on "Approval required" for this app registration), a bot registration, and typically a paid Azure Communication Services resource for the media/transcription pipeline. This is a multi-week Azure admin project, not a code change, and nothing in this codebase gets us meaningfully closer today.

What we already have instead: the canvas's existing live transcript dock (#transcript / startRec/stopRec in canvasApp.js) uses the browser's own Web Speech API on the local microphone — real, working, already attributed to the signed-in user. It has no access to Teams' audio pipeline directly; it would only pick up Teams call audio incidentally if Teams' own output were audibly playing through the same computer's speakers into that same open mic (unreliable, not a supported design). Recommendation: build Phase 2's "live capture" around this existing mic-based transcript, not a Teams-native feed.

Recommended new agents

  • extract_requirements — the single highest-value new agent; turns live transcript + ingested docs into structured, cited REQ cards.
  • capability_map — groups requirements/findings into the 4-group capability taxonomy with maturity + opportunity scoring.

Both would follow the exact draft→JSON-contract→coerce→persist pattern already proven twice this session (deepresearch, workflow) — this is a template now, not a one-off.
