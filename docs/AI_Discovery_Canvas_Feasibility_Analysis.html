1. Executive Summary

Bottom line: NaviCORE is a strong backbone for AI Discovery Canvas — for the AI/data layer, not for the canvas UI itself. The backend already has almost everything a BA-facing agent catalogue needs (Azure OpenAI calling convention, a RAG ingestion/retrieval pipeline, a structured "draft JSON → human review" generation pattern, a Neo4j graph capable of modeling provenance, an OAuth connector pattern, guardrails for regulated data). The existing frontend shell (auth, routing, design tokens, icon system, a lens/segmented-control pattern, an OAuth "Connect" component, SSE-driven live status, a chat-to-fill panel) is also directly reusable scaffolding.

What does not exist anywhere in the codebase is an actual infinite-canvas whiteboard engine — pan/zoom via transform, a minimap, freeform stickies/shapes/frames, multi-object drag, presentation mode, or a draft-card Approve/Edit/Reject lifecycle. The closest analog, backend/static/legacy_atlas.html (the Workflow Builder), is a 14,656-line, framework-free, single-purpose pipeline editor that the codebase's own migration notes are actively trying to retire — extending it further would fight the direction the codebase is already moving in, not help it.

Also entirely absent: Microsoft Teams / Graph API code, draw.io code, Granola code, and any speech-to-text/transcription code. These are genuine net-new integrations, though the existing OAuth connector pattern (source_oauth.py + ConnectProvider.jsx) is a clean template to build them on.

Recommendation in one line: stand up AI Discovery Canvas as its own project folder next to frd-generator (per the plan already in mind), copy over the reusable backend services and frontend shell pieces named in Section 10, build the canvas itself fresh in React on top of a library like React Flow, and treat Teams/Granola/draw.io as Phase 2 integrations gated by real client demand (see Section 12).

2. What We're Building — Target Product Recap

Per the Master Documentation and the interactive prototype you shared, AI Discovery Canvas is a single infinite whiteboard for Business Analysts, organised into four zones — Prepare, Run, Synthesize, Project — with a context-aware AI assistant that performs the BA's recurring tasks and files every output into a structured, provenance-tagged artifact library. Rather than repeat the full spec here, this section only lists the concrete UI/technical surface the reuse analysis in Sections 4–9 is measured against:

Canvas mechanics
True infinite pan/zoom (transform-based, not scrollbars), a minimap with click-to-fly, "fit everything," region/zone dashed borders, Focus (dim other zones) toggle.
Node types
Sticky notes, rectangles/ellipses, free text, connectors, frames, and structured "card" nodes (AI-generated artifacts) — each independently draggable/resizable/colorable via a node context toolbar.
Assistant panel
Chat thread, slash-command agent catalogue, "Suggested next" contextual actions, zone-aware context chip, file attach, web-research toggle.
Draft lifecycle
Every agent output renders as a draft card with Approve / Edit / Reject; only Approve places a canvas node and files the artifact.
Artifact library
Left drawer, 6 auto-organising folders (Background, How it works, Requirements, Issues & decisions, Meeting notes, Proposal), each item carrying a provenance tag (zone, source, approver, timestamp).
Live capture
Floating transcript dock with live pulse indicator, one-click Summarize; optional Microsoft Teams meeting link; optional Granola pre-workshop notes.
Facilitation chips
Timer, Vote, Record, Present (a chrome-free stakeholder walkthrough stepping through the four zones).
Search & traceability
Full-text search across all canvas nodes with match-count and auto-pan/zoom to the first hit.
Diagramming
"Draw process flow" agent round-trips a native .drawio file via the free embeddable draw.io editor.
Export
Handoff modal assembling process flows, stories, acceptance criteria, docs, MoM, and proposal into one DOCX package (PDF/Markdown planned).

19 agents across the four zones (Ingest client docs, Research company, Context brief, Questions to ask, Draft agenda — Prepare; Transcribe, Summarize, Draw process flow, Find gaps, Capture decisions — Run; User stories, Acceptance criteria, Documentation, Find opportunities, Minutes of Meeting — Synthesize; Draft SOW, Calculate ROI, Benefit-vs-risk, Suggest team — Project) are the unit this analysis maps against NaviCORE's existing capabilities in Section 5.

Four "products" already live in this one app, all behind the same auth/shell:

  • Skills landing (/skills) — a card-grid picking one of the app's workflows.
  • Atlas / Workflow Builder (/atlas) — a legacy, iframed, 14.6k-line monolith: a source→agent→output pipeline canvas, plus a separate vis-network "Knowledge Atlas" graph explorer with its own chat assistant.

The codebase's own CLAUDE.md describes the Atlas iframe as a deliberate strangler-fig seam — the explicit direction is to keep extracting panels into real React components over time, not to keep adding to the monolith. That direction matters directly for the canvas-engine decision in Section 9.

4. Component-Level Reuse Map

4.1 Canvas / Workflow Builder engine (backend/static/legacy_atlas.html)

14,656 lines · no framework · global-scope JS
Directly reusable
Node/edge JSON model ({id,type,kind,x,y,status,connected,config} / {id,from,to}); collision-proof id generation; SVG cubic-bezier connector math with 1→many fan-out; whole-node-box drop targets; debounced autosave-to-backend; a generic streamSSE() consumer; toast/modal chrome; a scope-aware chat dock shell.
Needs rework
Node kinds are hardcoded to source/agent/output/orchestration (a domain-specific pipeline, not a general whiteboard); the canvas viewport is native overflow:auto scrolling, not a real transform-based zoom/pan — there is no infinite-canvas primitive here at all, even in the separate vis-network "galaxy" pane, which has its own bolted-on zoom widget but is a different library entirely.
Missing entirely
Minimap, frames/regions/grouping, sticky notes/shapes as first-class objects, slash-commands, Approve/Edit/Reject draft-card lifecycle, live transcript dock, presentation mode, artifact-library drawer, search-and-fly-to-node, per-node color swatches.
Verdict: mine it for interaction ideas and the connector/autosave/SSE patterns; do not extend it in place. Roughly a quarter of the canvas-engine effort (data-model shape, connector geometry, save/run plumbing) is reusable as design reference — the majority of the surface (zoom/pan, minimap, stickies, frames, presentation mode, draft cards, artifact drawer) is net-new regardless of starting point, and building it inside a 14k-line unframeworked file is strictly more expensive than building it fresh in React.

4.2 Agent / LLM generation pattern

LLM calling
backend/app/services/llm_service.py::complete() — a single Azure OpenAI GPT-4.1 entry point: rate-limited, retried, vision-capable, context-truncating. Generic enough to back every new agent unchanged.
Draft→approve template
agentic_analyzer.py and impact_generator.py both follow: prompt → strict JSON schema → _coerce() validation/clamping → deterministic no-LLM fallback. This is exactly the "produce a structured draft, let a human approve" shape the Discovery Canvas needs for every agent output.
Multi-perspective generation
impact_generator.py's 3-lens (Business/Tech/Execution) KPI+narrative pattern is close to a 1:1 template for "Benefit vs. risk," and disrupt.py's quantified-KPI lens generation is a strong template for "Calculate ROI."
Verdict: this is the single most reusable layer in the whole codebase for this project — it is effectively a JSON-draft generation SDK already, just not packaged as one.

4.3 Ingestion & RAG

Pipeline
backend/app/services/rag/ — chunking (chunking.py), embedding via text-embedding-3-large (embedder.py), FAISS storage (store.py), scoped retrieval (service.py), per-source content-fingerprint caching and incremental re-index (workflow_sources.py, 871 lines).
File types
file_extractor.py already handles PDF, DOCX, XLSX, CSV, HTML, ZIP — directly covers the "Ingest client docs" requirement (PDF/DOCX/XLSX).
Summarization
code_summarizer.py's chunk→rollup→cache pattern generalizes cleanly to "Context brief" and "Summarize."
Verdict: high reuse. "Research company" (public web info) is the one gap here — no web-search connector exists anywhere in the backend; it needs a new fetcher, though it slots into the same RAG retrieval/LLM pipeline once fetched.

4.4 Neo4j graph & artifact/provenance modeling

Existing model
database/schema.py, knowledge_graph.py, workflow_assets.py already model Source → Summary → GeneratedDoc lineage via relationships like CONTRIBUTED_BY, PRODUCED_BY, RUN_FOR_PROJECT, plus a "latest version" pattern for capability maps. schema.py even reserves an unpopulated :UserStory label already.
What's needed
A provenance property set {source, zone, approver, timestamp} is additive to this model, not a redesign. New labels for Zone, Artifact, DraftCard, and Engagement (see Section 11) layer on top of the existing lineage pattern.
Verdict: adapt, don't replace — the graph's existing lineage-modeling philosophy is exactly what "traceability" (FR-09 in the target spec) needs.

4.5 Auth, multi-tenancy & OAuth connectors

Auth
backend/app/auth/ — mock provider and Azure/Entra ID (validated JWT via JWKS) both exist and work today. Everything currently scopes by project_id/kg_id, not by organisation/engagement — usable as an engagement-scoping seam with moderate adaptation, not a new subsystem.
OAuth connector pattern (backend)
source_oauth.py — GitHub + Azure DevOps via Authorization Code + PKCE, tokens referenced through Key Vault. Clean template for adding Microsoft Graph (Teams) as a third provider.
OAuth connector pattern (frontend)
frontend/src/shared/ConnectProvider.jsx is a fully generic, already-built "Connect X" component: POST /connections/<provider>/start → popup → postMessage callback → caller stores only a connection_id, never a raw token. sources.js re-exports a shared source catalog (CATALOG_SOURCES, PROVIDER_DISPLAY, etc.) used by both the legacy Workflow Builder and the React shell.
Verdict: this is the strongest existing foundation for the Teams/Granola/draw.io "Connect" UI in Section 6 — add new provider entries to one catalog file and reuse the existing popup-OAuth component as-is.

4.6 Guardrails & compliance

Existing
backend/app/services/guardrails/ — PII/secret masking before embedding (masker, NER detector, regex detectors), mode-based policy, an audit trail, and a vault. Directly relevant to the target's NFR around regulated-industry (pharma/healthcare) data handling.
Verdict: reusable as-is for masking transcripts/documents before they hit embeddings or third-party integrations — a real asset given the target document's own "Before You Build Anything" consent/compliance warning.

4.7 Frontend shell, design system & interaction patterns

App shell
App.jsx — React Router v6, everything wrapped in <RequireAuth>; adding a new top-level route (e.g. /canvas) is a same-pattern addition.
Icon system
components/Icon.jsx — one GLYPHS map of inline-SVG stroke paths keyed by name, paint via currentColor — architecturally identical to the icon dictionary the prototype itself uses.
Zone/lens pattern
shared/LensContext.jsx + LensSwitcher.jsx — a Context + segmented-control + persisted-to-localStorage pattern, currently projecting one analysis onto Business/Tech/Execution personas. The mechanism (not the specific lenses) maps directly onto Prepare/Run/Synthesize/Project — recommend a sibling ZoneContext rather than repurposing the persona lens itself, since zones need independent per-zone canvas state, not just a content projection.
Card-grid pattern
SkillsPage.jsx's data-array → clickable card (icon/title/desc/CTA/disabled-state) reuses directly for an agent catalogue or engagement picker.
SSE live-status
readSSE() in ModernizePage.jsx/DisruptPage.jsx — an async-generator SSE consumer used for Analyze/Impact/Pipeline/Deep-Dive — reusable for streaming draft-card generation.
Chat-to-fill panel
Disrupt's chat intent-parser panel (greeting, bubbles, typing indicator, POST-to-parse-intent) is the closest existing analog to the Assistant chat drawer's composer.
Design tokens
Dark theme via CSS custom properties in global.css (--bg/--panel/--accent/--cyan…) plus consistent card/chip/segmented-control/drawer conventions across feature CSS files — a solid visual-consistency base, though the prototype you shared is a light theme, so tokens would need a light variant or the two would need reconciling.
Verdict: very high reuse for the application shell; only the canvas stage itself falls outside this system.

4.8 Export

Existing
impact_generator.export_html() — server renders HTML, frontend opens it and calls window.print(). This is HTML-to-PDF via the browser print dialog, not a server-side generator.
Gap
No DOCX-generation library is installed (only python-docx for reading), no PDF library (reportlab/weasyprint) is installed. FR-11 requires a real DOCX export of the consolidated package — this needs a new dependency and a new export module.

6. Integrations Feasibility (draw.io, Microsoft Teams, Granola)

Confirmed by direct search across backend/: zero existing code for Microsoft Graph API/Teams, draw.io, Granola, or any speech-to-text/transcription library. All three (four, counting transcription) are genuine net-new work — but the effort differs a lot per integration:

Lowest effort

draw.io

No subscription, no OAuth. Needs: a small library to read/write .drawio XML, and the free embeddable draw.io editor dropped into the "edit this properly" flow. Nothing in NaviCORE helps or hinders this — it's a self-contained, low-risk first integration, exactly as the target doc's own suggested build order says.

Moderate effort, reusable OAuth shape

Microsoft Teams (Graph API)

No Graph SDK (msal/msgraph-sdk) is installed and no Teams code exists — but source_oauth.py's Authorization Code + PKCE + Key Vault-referenced token pattern is a proven template already exercised for GitHub/Azure DevOps, and the frontend ConnectProvider.jsx popup-OAuth flow needs zero new frontend code beyond a new catalog entry. The org almost certainly already has Microsoft 365, so this is a one-time technical setup (app registration + client IT approval), not new ongoing cost — matching the target document's own assessment.

Same OAuth shape, ongoing cost

Granola

Same reusable OAuth-connector shape as Teams (personal API or MCP, both need the Business plan). No transcription-quality gain over building your own — Granola is valuable specifically for its casual pre-workshop meeting-notes UX, not a capability gap NaviCORE fills elsewhere. Confirms the target document's own guidance: build this last, only if the pre-workshop-call use case justifies the ~$14/seat/month.

Genuinely new capability

Transcription (in-app microphone)

Not an "integration" in the OAuth sense — needs a speech-to-text engine (e.g. Azure Speech, Whisper) wired into a new live-audio pipeline feeding the same transcript store the Teams Graph API path would also feed. No existing NaviCORE code touches audio at all; budget this as the single largest net-new backend subsystem in the whole project.

Table/Section 2: The existing source_oauth.py + ConnectProvider.jsx pair should be treated as the one integration foundation worth generalizing early — every third-party connector (Teams, Granola, and eventually anything else) rides the same rails.

7. Gap Analysis — Net-New Work Required

Consolidating Sections 4–6, the concrete list of things that must be built with no meaningful existing analog:

  • Infinite-canvas engine — real transform-based pan/zoom, minimap, "fit everything," dashed-border zone regions, Focus dim-toggle. (See Section 9 for the recommended approach.)
  • Freeform node types — sticky notes, rectangles/ellipses, free text, frames, each draggable/resizable/colorable via a node context toolbar.
  • Draft-card Approve/Edit/Reject lifecycle — a first-class object type distinct from a chat message, with three explicit terminal actions and only Approve committing to canvas + artifact library.
  • Artifact library drawer — persistent, cross-session, six-folder structure with provenance tags, distinct from the Atlas's project-scoped "doc bar."
  • Slash-command agent catalogue in chat, plus a "Suggested next" contextual action list keyed to the active zone.
  • Full-text search across canvas nodes with match count and auto-pan/zoom-to-hit.
  • Live transcript dock + underlying speech-to-text pipeline (in-app mic) and/or Teams Graph API live-transcript streaming.
  • Presentation Mode — chrome-free, steps through the four zones with Previous/Next/Exit.
  • Server-side DOCX export of the consolidated handoff package (current export is HTML→browser-print only).
  • draw.io round-trip, Teams Graph API connector, Granola connector — all zero-code today (Section 6).
  • Engagement-level multi-tenant scoping — current scoping is per-project/per-kg, not per-client-organisation with its own access control.
  • Suggest team / staffing agent — no skills or staffing data model exists to ground this against.

8. Standalone Project vs. New Module — the Decision

You proposed creating a new folder next to frd-generator and copying the reusable pieces across, rather than adding AI Discovery Canvas as a fifth product inside the existing app. Both are legitimate; here's the actual trade-off, grounded in what Section 4 found:

New standalone folder (your plan)New route inside frd-generator
Product separationClean — ships/sells/versions independently of Modernize/Disrupt/AtlasCoupled — one deploy, one release train, harder to spin off later
Shared auth/org dataDuplicated unless you factor out a shared auth serviceFree — same Entra ID tenant, same session, same org model
Reuse mechanismCopy-and-diverge (Section 10) — reused code can drift out of sync with upstream fixesReuse-in-place — bug fixes to llm_service.py/RAG benefit both products automatically
Design systemCopy CSS tokens/Icon.jsx once, then free to diverge into the prototype's light themeMust reconcile the existing dark theme with the target's light-themed prototype
Migration-path fitMatches "canvas needs a real framework, stop extending the monolith" finding in 4.1Also fits — a new route doesn't touch the monolith either
Recommendation: your instinct to create a separate project is reasonable given how different a whiteboard product is from Modernize/Disrupt/Atlas, and it avoids reconciling two visual design languages. The one thing worth deciding explicitly up front (not by default) is auth: if AI Discovery Canvas will ever need to share users/organisations with NaviCORE, factor backend/app/auth/ out as a shared service now rather than forking it — forking auth is the one copy that's expensive to un-fork later. Everything else in Section 10's copy list is safe to fork-and-diverge.

9. Recommended Canvas Engine

No canvas library exists in the codebase today (the closest analog, vis-network, is a graph-layout library, not a whiteboard). Three off-the-shelf options were weighed against the target feature list:

LibraryFitTrade-off
React FlowRecommendedNative pan/zoom/minimap/node-and-edge model out of the box — closest match to the target's node/connector/region needs; less native support for true freeform sticky-note drawing, but that's a thin layer on top of its custom-node API
tldrawStrong alternativeBest-in-class freeform whiteboard (stickies, shapes, frames, presentation-style camera moves) out of the box; weaker for the structured "card with Approve/Edit/Reject + provenance" object model, which would need to be built as a custom shape type
Konva / custom SVGNot recommendedMaximum control, but re-implements minimap/pan-zoom/hit-testing from scratch — closer in cost to extending legacy_atlas.html's approach than to using a library

Given the target UI mixes structured pipeline-like zones (Prepare→Run→Synthesize→Project, agent-produced cards) with freeform annotation (stickies/shapes/frames), a pragmatic path is React Flow as the base (regions, cards, connectors, minimap, zoom) with a small custom "annotation" node type layer for stickies/shapes/text — rather than adopting a pure whiteboard library and fighting it to express structured zones and provenance-tagged cards.

12. Phased Build Roadmap

Phase 0 — Fork the foundation (days, not weeks)

  • Create the sibling project; copy the modules marked Reuse/copy in Sections 4 & 10; stand up Neo4j schema extensions from Section 11.

Phase 1 — Canvas skeleton

  • React Flow-based Prepare/Run/Synthesize/Project regions, pan/zoom/minimap, sticky/shape/frame node types, node context toolbar, Focus toggle.

Phase 2 — Assistant & agents

  • Chat drawer + slash-commands + Suggested Next; wire the 6 "Reuse" and 6 "Adapt" agents first (fastest path to a convincing demo); draft-card Approve/Edit/Reject; artifact library drawer with provenance.

Phase 3 — Live capture

  • In-app mic transcription (new STT subsystem), live transcript dock, remaining "Build" agents (Questions, Agenda, Acceptance Criteria, Capture decisions, Suggest team).

  • draw.io round-trip (lowest effort) → Microsoft Teams Graph API (reuses OAuth pattern) → Granola (only if pre-workshop-call use case justifies cost) → server-side DOCX export → Presentation Mode → search-and-fly-to-node.
